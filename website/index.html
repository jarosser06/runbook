Runbook

One config. Two audiences.

Runbook wraps your existing tools (make, npm, docker, custom scripts) and exposes them to humans and AI through the same running process. When you run runbook run build and your AI calls the build MCP tool, they execute the exact same command with the exact same flags. No drift. No guessing. No forgotten steps.

Get Started GitHub

How it works

Most projects have workflows scattered across Makefiles, npm scripts, shell scripts, and Docker commands, plus a separate set of AI prompts that describe the same things in different words. Runbook eliminates that split without replacing any of your tools. Define once in YAML, point it at your existing commands, and both audiences use the same definitions through the same running process.

Human developer

  • Runs runbook run build from terminal
  • CLI auto-proxies through running server
  • Manages daemons with start/stop/logs

Quick Start

1
Step 1

Create a config directory

Create a .runbook/ directory at the root of your project with one or more YAML files. Point tasks at whatever tools you already use. Runbook wraps them, it doesn't replace them.

.runbook/tasks.yaml 
version: "1.0"

defaults:
  shell: "/bin/bash"
  timeout: 300

task_groups:
  backend:
    description: "Go API tasks"
    tasks: [build, test, lint]
  frontend:
    description: "React app tasks"
    tasks: [frontend-install, frontend-build]

tasks:
  # Backend (wraps Make)
  build:
    description: "Compile the API binary"
    command: "make build"
    type: oneshot
    depends_on: [lint]

  test:
    description: "Run Go test suite"
    command: "make test"
    type: oneshot

  lint:
    description: "Run Go linter"
    command: "make lint"
    type: oneshot

  # Frontend (wraps npm)
  frontend-install:
    description: "Install frontend dependencies"
    command: "npm install"
    type: oneshot
    working_directory: "./frontend"

  frontend-build:
    description: "Build frontend for production"
    command: "npm run build"
    type: oneshot
    working_directory: "./frontend"

  # Infrastructure (wraps Docker)
  infra-up:
    description: "Start Postgres and Redis"
    command: "docker compose up -d"
    type: oneshot

  # Deploy (custom script with parameters)
  deploy:
    description: "Deploy to an environment"
    command: "scripts/deploy.sh --env {{.env}} --version {{.version}}"
    type: oneshot
    parameters:
      env:
        type: string
        required: true
        description: "Target environment (staging, production)"
      version:
        type: string
        required: false
        description: "Version tag to deploy"
        default: "latest"

  # Daemons (long-running dev servers)
  api:
    description: "API dev server with hot reload"
    command: "make dev"
    type: daemon

  frontend-dev:
    description: "Frontend dev server"
    command: "npm run dev"
    type: daemon
    working_directory: "./frontend"

workflows:
  ci:
    description: "Lint, test, and build"
    steps:
      - task: lint
      - task: test
      - task: build

  setup:
    description: "Bootstrap local dev environment"
    steps:
      - task: infra-up
      - task: frontend-install

prompts:
  dev-workflow:
    description: "How to develop and test this project"
    content: |
      Development workflow:
      1. Bootstrap: run the setup workflow
      2. Start servers: {{.Tasks.api.Start}} and {{.Tasks.frontend-dev.Start}}
      3. Before committing: {{.Tasks.ci.Run}}
      4. Deploy to staging: {{.Tasks.deploy.Run}} with env=staging

resources:
  architecture:
    description: "Project structure and tooling overview"
    content: |
      Backend: Go API, managed via Makefile
      Frontend: React SPA in ./frontend, managed via npm
      Infrastructure: Postgres + Redis, managed via docker compose
2
Step 2

Run tasks from the terminal

Use the CLI subcommands to run tasks directly. No AI required.

# See what tasks are available
runbook list

# These call make, npm, docker, etc.
runbook run lint
runbook run test
runbook run build

# Run with parameters
runbook run deploy --env=staging

# Start a daemon, check status, view logs, stop it
runbook start api
runbook status api
runbook logs api --lines=50
runbook stop api

Step 4

Ask your AI assistant

Your tasks are now MCP tools. Ask your AI to run them, or let it read the prompt you defined to understand the full workflow.

"Run the CI workflow"
"Start the API and frontend dev servers"
"Deploy version 1.4.2 to staging"
"Show me the API server logs"

Shared MCP server

By default Runbook runs in stdio mode. One process per AI session, no shared state. HTTP mode changes that: a single server runs continuously and every client connects to the same process. When a server is running, CLI commands automatically detect it and proxy through it, so your terminal and every AI session are hitting the same running process.

# Start once, connect from anywhere
runbook serve

# or on a custom port
runbook serve --addr :9000

Point each MCP client at http://localhost:9000. All sessions share the same tasks, running daemons, and log history. runbook run build from your terminal proxies through the same server the AI is using.

When multiple AI sessions trigger the same one-shot task with identical parameters simultaneously, Runbook deduplicates the execution. One run happens and both sessions receive the result. No redundant builds, no wasted compute.

Why this matters

Stdio mode starts a new process per client. HTTP mode runs one process all clients share.

All three see the same daemon state. If one session starts api, another can check its logs or stop it. The refresh_config MCP tool lets any session reload the config without restarting the server. Running daemons are not disrupted.

Task Types

Every task has a type that controls how Runbook manages its lifecycle. Tasks can also be marked disable_mcp: true to make them CLI-only. Useful for internal or privileged tasks you want humans to run explicitly but don't want exposed to AI sessions.

Oneshot

type: oneshot

Runs a command and returns the output when it exits. Use for build, test, lint, format. Anything that wraps an existing tool with a clear beginning and end.

tasks:
  test:
    description: "Run test suite"
    command: "make test"
    type: oneshot
    depends_on: [lint]
    timeout: 120

Daemon

type: daemon

Starts a long-running background process. Runbook registers separate start, stop, status, and logs MCP tools for each daemon automatically.

tasks:
  api:
    description: "API dev server"
    command: "make dev"
    type: daemon
    working_directory: "/path/to/project"

Parameters

Tasks accept parameters using Go template syntax. The AI fills in values automatically based on the parameter schema you define. Each parameter becomes a field in the MCP tool schema with its description shown to the AI.

tasks:
  deploy:
    description: "Deploy to an environment"
    command: "scripts/deploy.sh --env {{.env}} --version {{.version}}"
    type: oneshot
    parameters:
      env:
        type: string
        required: true
        description: "Target environment (staging, production)"
      version:
        type: string
        required: false
        description: "Version to deploy"
        default: "latest"

Set expose_working_directory: true to add a working_directory parameter to the MCP tool at runtime. This makes tasks reusable across different project paths without hardcoding a directory in the config.

Workflows

Workflows run multiple tasks in sequence with a single MCP tool call. Steps share optional parameters via template substitution, and individual steps can be marked continue_on_failure to keep the pipeline running even if they fail. Use depends_on on individual tasks to declare dependencies that are resolved automatically before the task runs, even when called outside a workflow.

workflows:
  ci:
    description: "Full CI pipeline"
    steps:
      - task: lint
      - task: test
      - task: build
        continue_on_failure: true

Runbook exposes each workflow as a single MCP tool named run_workflow_{name}. The AI can run your entire CI pipeline with one call.

MCP Prompts

Prompts are templated text resources that give AI assistants a playbook for your project. Not just what tools exist, but how to use them, in what order, and for what purpose. Define them under the prompts key using Go template syntax to reference your tasks by name.

prompts:
  dev-workflow:
    description: "How to develop and test this project"
    content: |
      Development workflow for this project:

      1. Bootstrap the environment: run the setup workflow
      2. Start backend: {{.Tasks.api.Start}}
      3. Start frontend: {{run_task "frontend-dev"}}
      4. Run before committing: {{.Tasks.ci.Run}}
      5. Deploy to staging: {{.Tasks.deploy.Run}} with env=staging

      To check server health: {{.Tasks.api.Status}}
      To tail server logs:   {{.Tasks.api.Logs}}

The template variables resolve to actual MCP tool names. {{.Tasks.api.Start}} becomes start_api, {{.Tasks.ci.Run}} becomes run_workflow_ci. The AI receives concrete tool names it can call directly, not vague descriptions.

For task names containing hyphens, use the run_task helper ‚Äî Go's template engine does not allow hyphens in dot-access chains. {{run_task "my-task"}} resolves to run_my-task and works with any task name regardless of what characters it contains.

Available template helpers: run_task "name". Dot-access methods per task: .Run (oneshot), .Start / .Stop / .Status / .Logs (daemon), .Desc (description text). These work for task names without hyphens.

MCP Resources

Runbook exposes several built-in MCP resources that give AI deep context about your project structure:

URIDescription
dev-workflow://task-groupsAll task groups and their member tasks, as JSON.
dev-workflow://task-dependenciesTask dependency graph showing which tasks depend on which others.
dev-workflow://docs/templatesComplete guide to prompt and command template syntax.
dev-workflow://docs/configurationFull configuration reference with examples.

Add your own custom resources under the resources key with either inline content or a file path. Custom resources also support template variables, so you can reference task names dynamically.

resources:
  architecture:
    description: "Project structure and tooling overview"
    content: |
      Backend: Go API in ./cmd/server, managed via Makefile
      Frontend: React SPA in ./frontend, managed via npm
      Infrastructure: Postgres + Redis, managed via docker compose

  deployment-guide:
    description: "How to deploy this service"
    file: "./docs/deployment.md"
    mime_type: "text/markdown"

Custom resources are mounted at runbook://custom/{name} and are accessible to any AI session connected to the server.

Config Reference

Top-level fields

FieldDescription
versionConfig version. Use "1.0".
importsList of glob patterns to merge additional config files. All merged at startup; task names must be unique across files.
defaultsDefault timeout (seconds), shell, and env applied to every task unless overridden.
tasksMap of named tasks. Each key becomes an MCP tool name.
workflowsMap of named multi-step workflows that chain tasks sequentially.
task_groupsNamed collections of tasks with a description and tasks list. Exposed via the dev-workflow://task-groups MCP resource.
promptsTemplated prompts exposed as MCP prompt resources. Use Go template syntax to reference task tool names dynamically.
resourcesCustom MCP resources with inline content or a file path. Optionally set mime_type.

Overrides file

Place a .runbook.overrides.yaml file at the project root to control task visibility without editing your config files. This is useful for locally disabling tasks shared across a team. Glob patterns are supported.

# .runbook.overrides.yaml
tasks:
  deploy:
    disabled: true       # fully disable
  ts-*:
    disable_mcp: true   # hide from AI, keep CLI access

workflows:
  ci:
    disabled: true

prompts:
  dev-workflow:
    disabled: true

resources:
  architecture:
    disabled: true

Overrides are loaded after the main config and are additive ‚Äî once a flag is set to true, it stays set. The overrides file is optional and is ignored if it does not exist.

Task fields

FieldDescription
descriptionHuman-readable description shown to the AI in the tool schema.
commandShell command to run. Can be any tool: make build, npm test, docker compose up -d, etc. Supports {{.param}} Go template substitution.
typeoneshot runs and exits. daemon is a long-running background process.
working_directorySet a fixed working directory for the command.
expose_working_directoryIf true, adds a working_directory parameter to the MCP tool so the caller can override it at runtime. Useful for reusable tasks across different project paths.
envMap of environment variables set for this task only.
timeoutTimeout in seconds. Overrides the value in defaults.
shellShell binary used to run the command (e.g. /bin/zsh). Overrides the value in defaults.
parametersNamed parameters with type, required, description, and optional default.
depends_onList of task names that must complete successfully before this task runs. Dependencies are resolved automatically regardless of how the task is invoked.
disable_mcpIf true, the task is excluded from MCP tool registration. It remains available via the CLI but is not visible to AI sessions.
disabledIf true, the task is fully disabled ‚Äî excluded from both MCP and CLI. Useful for temporarily hiding tasks without removing them from config.

CLI subcommands

CommandDescription
runbook initCreate a minimal .runbook/tasks.yaml in the current directory. Skips if it already exists. Also available as the init MCP tool so an AI can bootstrap the config from within a session.
runbook serve [--addr <addr>]Run as a standalone HTTP server instead of stdio MCP mode. Enables shared state, CLI auto-proxy, and execution deduplication across sessions. Default address: :8080.
runbook listList all available tasks and workflows with their types, descriptions, and parameters.
runbook run <task> [--param=value...]Run a oneshot task or workflow. Parameters passed as --key=value flags.
runbook start <task> [--param=value...]Start a daemon task in the background.
runbook stop <task>Stop a running daemon.
runbook status <task>Show whether a daemon is running and its PID.
runbook logs <task> [--lines=N] [--filter=REGEX] [--session=ID]Print task log output. Defaults to the latest session.

All subcommands accept --config=<path> to specify a config file or directory. When a server is running in HTTP mode, commands automatically detect and proxy through it.

Global flags

FlagDescription
--config <path>Path to a config file or directory. If omitted, Runbook looks for a .runbook/ directory in the current directory.
--working-dir <path>Set the project working directory before loading config and running commands.
--localBypass any running HTTP server and execute locally. Useful when you want to run a command in the current process without proxying.

The runbook serve subcommand accepts an additional --addr <addr> flag to set the listen address (default: :8080). Use runbook init to scaffold a starter .runbook/tasks.yaml in the current directory.

Built-in MCP tools

ToolDescription
initInitialize a .runbook/tasks.yaml from within an AI session. Accepts optional path and overwrite parameters.
refresh_configReload all configuration from disk without restarting the server. Re-registers tools, resources, and prompts. Running daemons are not disrupted. Lets AI pick up config edits immediately.
